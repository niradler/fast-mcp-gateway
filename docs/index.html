Why a gateway

Point an LLM at a dozen MCP servers directly and you get a dozen of every problem. The gateway puts one governed control plane in front of them all.

✕

Direct: a dozen of everything

  • ✕A dozen connections to manage
  • ✕A dozen auth schemes
  • ✕No namespacing, colliding tool names
  • ✕No central policy or audit
  • ✕No way to hide a dangerous tool
✓

Through the gateway

  • + One connection, many servers, each proxied under its own namespace (github_*, slack_*)
  • + Governed: filter tools, gate calls, redact results, audit everything, all as hooks
  • + Reuse, don't rebuild: FastMCP does proxying and transport bridging; we add the registry, groups, builder, and hook runner

Features

Thin core, everything cross-cutting is a hook or a plugin.

▤

Server registry (Store)

Persistent CRUD over upstream MCP servers. Ships with a zero-setup SqliteStore, swappable for Postgres / Redis / in-memory via one protocol.

⧉

Namespaced proxying

Each enabled server is mounted under its name as a prefix. reload() rebuilds the mounts from the registry.

⎇

Five hook seams

pre_mcp_connect, pre_list_tools, pre_tool_call, confirmation, post_tool_call. All plain async functions.

⊘

Access control

Per-server and per-group allow / deny glob lists, enforced on both list_tools (hides) and call_tool (blocks).

⊞

Groups & scoped endpoints

Expose a curated subset of servers/tools at /mcp/g/{group}, served by the same shared MCP app, no per-group duplication.

◧

Plugins

Bundle hooks, FastMCP middleware, an admin router, ASGI mounts, and meta-tools into one named extension with setup / teardown.

⚖

Optional policy engine

An agt extra wires Microsoft's agent-governance-toolkit (agent-os) in as a policy plugin: prompt-injection, semantic policy, response scan, egress.

↻

Hot reload

Register or change servers in the store, then call reload() (or POST /admin/reload): the gateway rebuilds its proxy mounts from the registry, no restart.

Architecture

The gateway is a parent FastMCP server that proxies each registered upstream under a namespace, exposed as an ASGI app you mount onto your own FastAPI app alongside an admin router for CRUD.

Quickstart

Create a gateway, mount it on FastAPI, register an upstream, reload.

1 · Mount the gateway
import os
from fastapi import FastAPI
from mcp_gateway import (
    ConnectContext, ConnectSettings, Hooks, SqliteStore, create_gateway,
)


async def inject_auth(ctx: ConnectContext) -> ConnectSettings | None:
    # Auth is just a hook: headers merged over the server's static headers.
    if ctx.server.name == "github":
        token = os.environ["GH_TOKEN"]
        return ConnectSettings(headers={"Authorization": f"Bearer {token}"})
    return None


gateway = create_gateway(
    store=SqliteStore("gateway.db"),
    hooks=Hooks(pre_mcp_connect=[inject_auth]),
)

app = FastAPI(lifespan=gateway.lifespan)
gateway.install(app)   # mounts /admin (CRUD) and /mcp (MCP endpoint)
2 · Register an upstream, then reload
curl -X POST http://127.0.0.1:8000/admin/servers \
  -H 'content-type: application/json' \
  -d '{"name": "math",
       "url": "http://127.0.0.1:9000/mcp/",
       "transport": "http"}'

curl -X POST http://127.0.0.1:8000/admin/reload
✓
Done
The upstream's tools now appear at the gateway under the math_ namespace.
Or run the bundled example
make run   # uvicorn examples.basic_app:app --reload
# Admin + OpenAPI docs: http://127.0.0.1:8000/docs
# MCP endpoint:         http://127.0.0.1:8000/mcp/

Everything is a hook

A hook is an async function in a Hooks container, passed at mount time. Each binds to the layer where it belongs and chains in registration order.

Hook Binds to Runs
pre_mcp_connectproxy client factorybefore opening an upstream session
pre_list_toolsHookMiddleware.on_list_toolson catalog requests
pre_tool_callon_call_tool (pre)before forwarding a call
confirmationon_call_tool (when REQUIRE_CONFIRMATION)human-in-the-loop approval
post_tool_callon_call_tool (post)after the upstream result
sequenceDiagram participant C as Client participant G as Gateway participant H as Confirmation hook participant U as Upstream MCP server C->>G: call_tool Note over G: pre_tool_call runs
continue / mutate args / deny / REQUIRE_CONFIRMATION G->>H: REQUIRE_CONFIRMATION Note over H: human approval
reject or none registered = denied H-->>G: approved G->>U: forward call (on continue) U-->>G: result Note over G: post_tool_call runs G-->>C: final result 
from mcp_gateway import Hooks, ToolCallResult, ToolDecision


async def block_deletes(ctx) -> ToolCallResult | None:
    if ctx.message.name.endswith("_delete_all"):
        return ToolCallResult(
            decision=ToolDecision.REQUIRE_CONFIRMATION,
            reason="destructive",
        )
    return None


async def approve(ctx) -> bool:
    return await ask_a_human(ctx.tool_name, ctx.arguments)


hooks = Hooks(pre_tool_call=[block_deletes], confirmation=[approve])
ℹ
Four outcomes per call
A pre_tool_call hook may continue, mutate args, deny, or return REQUIRE_CONFIRMATION, which triggers the confirmation hooks.
⚠
Confirmation is fail-safe
If any confirmation hook rejects, or none is registered, the call is denied. Policy, guardrails, audit, and cost limits are all just hooks, nothing special in the core.

Plugins bundle the cross-cutting stuff

Where a hook is one function, a plugin is a named bundle of extensions applied at create_gateway time. One plugin can contribute hooks and middleware and routes and tools at once.

Hooks

Functions merged into the gateway hook chain, all five seams.

FastMCP middleware

Around-the-call control like circuit breaking or retry that wraps execution itself.

Admin router

An APIRouter mounted under the admin prefix for the plugin.

ASGI mounts

(path, app) sub-apps mounted on the MCP ASGI app, e.g. a governance HTTP API.

Meta-tools

Register extra tools on the parent FastMCP server.

Lifecycle

Async setup / teardown bound to the gateway lifespan.

The Plugin protocol

A plugin implements a name, a contributions(context) method returning PluginContributions, and optional setup / teardown coroutines. The GatewayContext it receives exposes the store, the parent mcp, and a reload callable.

Pass plugins at create time
from mcp_gateway import create_gateway, SqliteStore

gateway = create_gateway(
    store=SqliteStore("gateway.db"),
    plugins=[MyAuditPlugin(), MyRateLimitPlugin()],
)
optional

The agent-os policy plugin

The agt extra wires Microsoft's agent-governance-toolkit (agent-os) in as AgtAgentOsPlugin. Its core capability is the policy engine: it evaluates policy for every tool call (scoped to the active group) and denies calls the policy rejects. Everything else is an opt-in toggle on AgtAgentOsSettings, which reuses agent-os config types DetectionConfig / IntentCategory / EgressRule.

Toggle Seam Effect
enable_prompt_injectionpre_tool_calldeny calls whose arguments look like prompt injection
enable_semantic_policy (+ semantic_deny)pre_tool_calldeny calls whose classified intent is dangerous
enable_response_scanpost_tool_callblock responses flagged unsafe (credential / PII / threat)
enable_credential_redactionpost_tool_callredact secrets / PII out of responses
enable_egress_policy (+ egress_rules)pre_mcp_connectrefuse upstreams whose URL is outside the allowlist
Install
pip install "fast-mcp-gateway[agt]"
Enable the toggles you want
from mcp_gateway import create_gateway, SqliteStore
from mcp_gateway.integrations.agt import AgtAgentOsPlugin, AgtAgentOsSettings

gateway = create_gateway(
    store=SqliteStore("gateway.db"),
    plugins=[
        AgtAgentOsPlugin(
            AgtAgentOsSettings(
                policy_dir="./policies",
                fail_closed=True,
                enable_prompt_injection=True,
                enable_response_scan=True,
                enable_credential_redaction=True,
            )
        )
    ],
)
